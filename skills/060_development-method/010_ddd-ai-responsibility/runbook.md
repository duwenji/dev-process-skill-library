# ddd-ai-responsibility Runbook

## 目的

DDD における AI と人の責任分担を、要件定義から運用まで一貫して整理する詳細手順書です。

## DDDとは？

ドメイン駆動設計（Domain-Driven Design, DDD）は、業務の本質をモデル化し、ソフトウェア設計と実装の中心に据えるアプローチです。開発者と業務担当者が共通言語で対話し、複雑な業務を説明できるモデルを育てることを重視します。

## 主要用語リスト

| 用語 | 説明 |
| ---- | ---- |
| ドメイン | 業務領域や問題領域そのもの |
| エンティティ | 識別子を持つ、業務上重要なオブジェクト |
| 値オブジェクト | 識別子を持たず、値で同一性が決まるオブジェクト |
| 集約 | 一貫性を保つためのエンティティと値オブジェクトのまとまり |
| リポジトリ | 集約の永続化・取得を担う仕組み |
| サービス | モデルに属さない操作や処理を表現するオブジェクト |
| ファクトリ | 複雑な生成処理を担うオブジェクト |
| ユビキタス言語 | 開発者と業務担当者が共有する共通言語 |
| バウンデッドコンテキスト | 用語やモデルの意味が一貫する境界 |
| ドメインイベント | 業務上重要な出来事を表すオブジェクト |

## Phase 1: 要件定義・ドメイン理解

### AIに任せられること（Phase 1）

- 過去資料・議事録・仕様書の要約
- 業務用語の整理・一覧化
- 業務フローのたたき台作成
- よくある要件・制約の列挙

### AIへの指示テンプレート（Phase 1）

この業務資料をもとに、業務用語、主要な業務フロー、制約条件、未確定事項を整理してください。解釈を足した箇所と、資料に根拠がある箇所を分けて示してください。

### 人が決めるべきこと（Phase 1）

- 業務の本質・価値の定義
- コア業務／補助業務の切り分け
- 採用する要件・捨てる要件
- 業務制約の優先順位
- Ubiquitous Language（共通言語）の最終決定

### 人がレビューすべき観点（Phase 1）

- 業務の目的と価値が正しく表現されているか
- 採用しない要件や例外条件が漏れていないか
- 用語の意味が現場の認識とずれていないか

## Phase 2: モデリング

### AIに任せられること（Phase 2）

- 名詞抽出によるエンティティ候補出し
- クラス図・PlantUMLのたたき台生成
- 制約条件のコード表現案

### AIへの指示テンプレート（Phase 2）

要件整理結果をもとに、エンティティ候補、値オブジェクト候補、集約候補、バウンデッドコンテキスト候補を挙げ、それぞれの根拠と迷う点も示してください。

### 人が決めるべきこと（Phase 2）

- エンティティ／値オブジェクトの判断
- Bounded Context の境界設定
- ドメインルールの所属先判断
- モデルが業務を正しく説明できているかの確認

### 人がレビューすべき観点（Phase 2）

- 識別子を持つべきものと値として扱うべきものが妥当か
- 境界の切り方が業務の責任範囲と一致しているか
- 重要な業務ルールの置き場所がぶれていないか

## Phase 3: 設計

### AIに任せられること（Phase 3）

- レイヤ構成・パッケージ構成例の提示
- DDL／API仕様のたたき台生成
- 例外パターンの洗い出し

### AIへの指示テンプレート（Phase 3）

このモデルを前提に、レイヤ構成、API案、データ設計案、例外時の扱いを整理してください。責務分離の意図と設計上のトレードオフも示してください。

### 人が決めるべきこと（Phase 3）

- 責務分離の妥当性判断
- 正規化／非正規化、性能設計のトレードオフ
- 公開インタフェースの安定性ポリシー
- 異常時に「どこで止めるか／継続するか」の判断

### 人がレビューすべき観点（Phase 3）

- 設計がモデルの意図を壊していないか
- 公開インタフェースが将来変更に耐えられるか
- 性能や運用の都合で無理な妥協をしていないか

## Phase 4: 実装

### AIに任せられること（Phase 4）

- CRUDや定型処理の実装
- リファクタリング案の提示
- コメント・説明文の自動生成

### AIへの指示テンプレート（Phase 4）

この設計に基づき、定型実装案と補助コード案を作成してください。ドメインルールが入る箇所と、単なる技術的処理の箇所を分けて示してください。

### 人が決めるべきこと（Phase 4）

- ドメインルールの配置場所
- リファクタ適用可否の判断
- 業務的な意味・背景の補足

### 人がレビューすべき観点（Phase 4）

- ドメインルールが不適切なレイヤに流れていないか
- 実装の都合で業務意味が削られていないか
- リファクタ案が可読性だけでなく意味の維持にも効いているか

## Phase 5: テスト・品質

### AIに任せられること（Phase 5）

- UT網羅テストケース生成
- IT/STシナリオ案の洗い出し
- 異常系・境界値パターン列挙

### AIへの指示テンプレート（Phase 5）

この仕様とモデルをもとに、正常系、異常系、境界値、重要ロジック中心のテスト観点とケース案を作成してください。重要度の高い順で整理してください。

### 人が決めるべきこと（Phase 5）

- 守るべき重要ロジックの優先順位
- 再発防止に効く観点の反映
- 実運用を踏まえた重大度判断

### 人がレビューすべき観点（Phase 5）

- 失敗時に困る業務ルールが優先的に検証されているか
- 再発防止につながる観点が含まれているか
- 実運用で起こりうる異常が机上の想定だけになっていないか

## Phase 6: 運用・進化

### AIに任せられること（Phase 6）

- ログ・アラートの要約
- 類似障害・過去事例の検索
- 影響範囲の整理

### AIへの指示テンプレート（Phase 6）

障害情報や運用ログをもとに、事象の整理、影響範囲、原因候補、改修案を分けて示してください。短期対処と中長期の改善案も区別してください。

### 人が決めるべきこと（Phase 6）

- 根本原因の解釈
- 改修・進化の方向性
- ドメインモデル変更の是非と覚悟

### 人がレビューすべき観点（Phase 6）

- 表面的な症状と根本原因を混同していないか
- 改修方針が今後のモデル進化と矛盾していないか
- モデル変更のコストと価値を引き受ける覚悟があるか

## 判定基準一覧

| 判定項目 | 基準 | 確認者 |
| --- | --- | --- |
| 要件整理 | 業務目的、制約、用語の認識差が整理されている | 人 |
| モデリング | 境界、責務、ルール配置が説明できる | 人 |
| 設計 | モデル意図と設計方針が矛盾しない | 人 |
| 実装 | ドメインルールの置き場所が妥当である | 人 |
| テスト | 重要ロジックと実運用リスクが優先されている | 人 |
| 運用・進化 | 根本原因と改善方針が切り分けられている | 人 |
