# api-contract-design Runbook

## 目的
API 契約の設計、互換性判断、エラー仕様、移行方針を整理する詳細手順書です。

## Phase 1: 契約論点整理・候補化

### 段階1: 準備（対象 API と利用者の収集）
作業:
1. 対象 API、利用者、依存先を整理する
2. 現行仕様、既存ドキュメント、実装を確認する
3. 公開範囲と互換性制約を確認する

### 段階2: 契約要件と互換性制約の詳細化
作業:
1. 必須入力、任意入力、出力を分類する
2. エラー応答と例外条件を整理する
3. 互換性ポリシーを明文化する

### 段階3: 開発者が契約条件を正式入力
入力テンプレート:
- API 名
- 利用者
- 入力
- 出力
- エラー
- 互換性要件
- 制約

### 段階4: AI が現行契約と依存先を分析
出力:
- 現行差分
- 利用側影響一覧
- 破壊的変更候補

### 段階5: AI が入出力フローを生成
出力:
- リクエスト処理フロー
- エラー分岐フロー
- バージョン分岐メモ

### 段階6: AI が契約案を生成
最低3案:
- 案1: 後方互換優先
- 案2: 明快な再設計
- 案3: 段階移行型

## Phase 2: 契約決定・仕様化

### 段階7: 契約案レビュー・決定
判定条件:
- 利用者影響を説明できるか
- エラー設計が明確か
- 互換性と実装コストのバランスが取れているか

### 段階8: AI が契約仕様を作成
含める項目:
- リクエスト、レスポンス
- エラーコード
- バージョニング
- 廃止予定項目
- 移行手順

### 段階9: 開発者が内容をレビュー・承認

### 段階12: AI が互換性と例外を整理
### 段階13: 契約最終レビュー・承認
