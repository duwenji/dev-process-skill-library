# 技術用語定義ガイド（Glossary）

このドキュメントは、dev-process-skill-library で使用される英語系技術用語の定義と使用方法を統一するための参考ページです。
**技術用語は英語のまま** で、日本語説明を提供しています。

省略用語（略語）の正本は本ドキュメントの「略語・日本語対応表」です。別ドキュメントで略語を使う場合は、必要に応じて本ページを参照してください。

---

## コア用語

### Skill（スキル）
**定義**：開発プロセスの特定のタスク（調査、設計、実装、テスト等）に対応した、段階的なワークフローテンプレート。  
**使用例**：`feature-implementation-unified`（新規機能・機能変更対応 Skill）  
**日本語での説明例**：「新規機能対応Skillの Phase 1 では要件を整理します」  

### Phase（フェーズ）
**定義**：Skill 内の段階的な処理ステップ（通常 Phase 1 ～ 4）。各 Phase は調査→分析→意思決定→実装 といった工程に対応。  
**使用例**：`Phase 1: Investigation（調査）`, `Phase 2: Design（設計）`  
**日本語での説明例**：「Phase 2 では複数の実装案を提示します」  

### Runbook（ラ ンブック）
**定義**：Skill の各 Phase ごとに提供される、実行手順と判定基準を記した詳細ガイド。  
**ファイル例**：`skills/design-and-implementation/feature-implementation-unified/phase1-investigation.md`  
**日本語での説明例**：「runbook に記載された判定ポイントを確認して Phase を進行させます」  

### Sub-Skills（サブスキル）
**定義**：Skill の一部項目（例：テスト戦略の特定パターン）に特化した補助テンプレート。  
**使用例**：`test-strategy-unified` の Sub-Skill として `unit-test-patterns.md`  
**日本語での説明例**：「このタスクに適用する Sub-Skill を選択してください」

## フロー・プロセス関連用語

### Gate Condition（ゲート条件）
**定義**：各 Phase の進行を承認するための条件・チェックリスト。通常は開発者による確認が必須。  
**使用例**：「Phase 2 進行のゲート条件：要件整理書が承認されていること」  
**表記統一**：`ゲート条件（開発者承認必須）` に統一  
**日本語での説明例**：「ゲート条件をすべて満たしているか確認してから Phase を進めてください」  

### Approval Flow（承認フロー）
**定義**：タスク・成果物・変更が承認者（Owner, Reviewer, Approver）を通じて承認される流れ。  
**ロール定義**：
- **Owner**：Skill の実行を主導する開発者
- **Reviewer**：成果物の内容を技術的に査閲する者
- **Approver**：最終承認を行う管理者・チームリード
- **Maintainer**：リポジトリ・Skill の保守者  
**日本語での説明例**：「Reviewer による査閲後、Approver の承認を得てください」  

### Balance Mode（バランスモード）
**定義**：Skill の実行モードの一つ。AI 支援と開発者の判断を適切に交える。  
**3つのモード**：
- `strict`：開発者の判断・入力がより多く必要（厳密・丁寧）
- `balance`：AI 支援と開発者判断のバランスを取る（推奨）
- `speed`：AI 支援により高速に進める（迅速・時間重視）  
**日本語での説明例**：「推奨モードは balance です。AI の代案を参考にしつつ、開発者の判断で選択してください」

## 形式・技術用語

### Frontmatter（フロントマッター）
**定義**：Markdown ファイルの冒頭に YAML 形式で記述されるメタデータ。ファイルの属性・用途を定義。  
**使用例**：
```yaml
---
name: feature-implementation-unified
description: 新規機能・機能変更対応Skill（統合フレームワーク）
user-invocable: true
disable-model-invocation: false
---
```
**日本語での説明例**：「Skill の frontmatter で、このテンプレートの目的と属性を指定します」  

### Kebab-case（ケバブケース）
**定義**：複数単語を小文字のハイフンで連結する命名規則（例：`feature-implementation-unified`）。  
**使用例**：ファイル名、フォルダ名、Skill 識別子  
**日本語での説明例**：「Skill フォルダ名は kebab-case で命名してください」  

### Mermaid（マーメイド）
**定義**：フロー図・クラス図などを Markdown で記述できる図表ライブラリ。  
**使用例**：`flowchart TD` による処理フローの図示  
**日本語での説明例**：「Mermaid 図で Phase の全体フローを表示します」  

### Append-only（追記のみ）
**定義**：ログ・履歴ファイルに対し、既存データの削除・上書きを禁止し、新規データのみを追加する運用方針。  
**目的**：監査証跡の完全性・改ざん防止  
**使用例**：`実行ログは append-only で記録し、既存履歴の削除・上書きは禁止します`  
**日本語での説明例**：「実行ログは append-only で記録することで、完全な履歴追跡可能性を確保します」  

### Receipt（レシート / 根拠記録）
**定義**：タスク実行時の選択・判定・決定理由を記録した根拠ドキュメント。  
**記録項目**：実行日時、判定者、判定根拠  
**日本語での説明例**：「の承認 receipt（進行記録）に、判定根拠を明記してください」  

### Acceptance Criteria（受入条件 / 受入基準）
**定義**：タスク・成果物が完成したと判断するための客観的な条件。  
**使用例**：「要件整理書が以下の受入条件を満たしていること：(1) 全要件が記載, (2) 優先度が設定, (3) 制約条件が明記」  
**日本語での説明例**：「この Phase を完了するために満たすべき受入条件は以下の通りです」

## その他の用語

### ERD（Entity-Relationship Diagram / ER図）
**定義**：データベーススキーマを表現する図。エンティティとその関連性を図示。  
**参考ページ**：`skills/shared-references/erd-best-practices.md`  
**日本語での説明例**：「DB 設計では ERD を作成して、テーブル構造を明確にします」  

### Diff（差分）
**定義**：ファイルやコード間の変更部分を抽出・表示したもの。version control で使用。  
**使用例**：Git の `git diff` コマンド、PR での差分表示  
**日本語での説明例**：「実装前後の diff をレビュアーに提示してください」  

### Non-Functional Requirements（非機能要件）
**定義**：パフォーマンス、セキュリティ、スケーラビリティなど、機能以外の要件。  
**使用例**：「API レスポンス時間 < 200ms」「暗号化通信対応」  
**日本語での説明例**：「新機能には以下の非機能要件を適用します」

## 略語・日本語対応表

### 責任・役割

### RACI
**英語**：Responsible, Accountable, Consulted, Informed  
**日本語**：責任分担マトリクス（実行責任、最終責任、相談先、報告先）  
**用途**：役割と責任を明確にする運用フレームワーク。

### PM
**英語**：Project Manager  
**日本語**：プロジェクトマネージャー  
**用途**：進行管理、優先度判断、意思決定支援を担う役割。

### QA
**英語**：Quality Assurance  
**日本語**：品質保証  
**用途**：品質基準、検証観点、受入判断の妥当性を確認する役割。

### 設計・実装

### ADR
**英語**：Architecture Decision Record  
**日本語**：設計判断記録  
**用途**：設計方針の採用理由、比較結果、却下理由を記録する文書。

### DDL
**英語**：Data Definition Language  
**日本語**：データ定義言語  
**用途**：テーブル、制約、インデックスなど DB スキーマを定義する SQL。

### 運用・計測

### KPI
**英語**：Key Performance Indicator  
**日本語**：主要業績評価指標  
**用途**：導入効果や運用品質を継続的に評価する指標。

### SLO
**英語**：Service Level Objective  
**日本語**：サービスレベル目標  
**用途**：可用性や遅延など、サービス品質目標を定量化する基準。

### TRK
**英語**：Tracking ID  
**日本語**：追跡識別子  
**用途**：実行ログやテスト項目を一意に追跡するための ID（例：`TRK-YYYY-MM-DD-NNN`）。

### EX
**英語**：Exception ID  
**日本語**：例外識別子  
**用途**：優先度例外などの運用例外を一意に管理するための ID（例：`EX-YYYY-MM-DD-NNN`）。

## 凡例・記号

### フロー図での役割表示（統一フォーマット）
- 🔵 **AI担当**：AI が自動または支援で実行するステップ
- 🟢 **開発者担当**：開発者による重要な入力・判定が必要なステップ
- 🟡⭐️ **ゲート条件（開発者承認必須）**：進行判定ポイント

**使用例**：
```
🔵 AI が要件差分を分析
🟢 開発者が複数案を検討し、実装方針を決定
🟡⭐️ ゲート条件：複数案の検討記録が approval を得ていること
```
