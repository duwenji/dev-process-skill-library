# data-model-design-unified Runbook

## 目的
要件から概念モデル、論理モデル、物理モデルを一貫して設計し、実装に渡せる形で確定する詳細手順書です。

## Phase 1: 要件整理・概念モデル抽出

### 段階1: 準備（業務境界と既存資産の収集）
作業:
1. 対象業務フロー、既存 DB、関連 API を整理する
2. 要件ID とユースケースを一覧化する
3. 既存制約（法令、監査、運用）を確認する

### 段階2: 要件とデータ制約の詳細化
作業:
1. 必須データと任意データを分類する
2. 整合性制約（一意性、参照整合、必須）を整理する
3. 期待性能（検索、更新頻度、保持期間）を確認する

### 段階3: 開発者がモデル設計条件を正式入力
入力テンプレート:
- 背景
- スコープ/非スコープ
- エンティティ候補
- 主要制約
- 性能要件
- 移行要否

### 段階4: AI が概念モデル候補を分析
出力:
- エンティティ候補
- 関係候補
- 競合/重複候補

### 段階5: AI が ERD と関係フローを生成
出力:
- 概念 ERD
- 主要関係フロー
- 不確定点メモ

### 段階6: AI がモデル案を生成
最低3案:
- 案1: 正規化優先
- 案2: 読み取り性能優先
- 案3: 段階移行優先

## Phase 2: 論理/物理モデル決定

### 段階7: モデル案レビュー・決定
判定条件:
- 要件IDとエンティティの対応が明確か
- カーディナリティと制約が説明可能か
- 運用コストが許容範囲か

### 段階8: AI が論理/物理モデルを作成
含める項目:
- テーブル定義
- PK/FK とインデックス
- 命名規則
- データ辞書
- DDL 方針

### 段階9: 開発者が内容をレビュー・承認
チェック:
- 実装可能性
- 監査要件との整合
- API 契約との整合

## Phase 3: 検証・移行計画確認

### 段階10: AI が検証観点を生成
含める項目:
- 正常系/異常系
- データ整合性
- 移行/ロールバック
- 監視観点

### 段階11: 観点レビュー・承認
### 段階12: AI が整合性と移行手順を整理
出力:
- 移行手順
- 互換性影響
- 例外一覧

### 段階13: モデル最終レビュー・承認
判定条件:
- データ辞書と DDL 方針が一致しているか
- 移行手順が現実的か
- 未確定事項が管理されているか

## Phase 4: 報告

### 段階14: AI がモデル確定内容を報告
出力:
- 採用モデル
- 移行計画
- 実装引き継ぎ項目
