# Structuring: 構造化

省略用語（RACI, KPI, ADR, DDL, SLO, QA, PM, TRK, EX）は [../../../shared-references/glossary.md](../../../shared-references/glossary.md) の『略語・日本語対応表』を参照してください。

## 目的

Intake で収集した知識を、標準化成果物へ変換できる形に分解・分類し、操作種別（新規 / 改善 / マージ / 削除）を決定する。

## 実施ポイント

1. 知識を「判断ルール / 手順 / 注意点 / 指標」の4種に分解する
2. `skills/` 配下の既存 Skill と照合し、重複・類似を検出する
3. 重複検出結果と再利用価値をもとに操作種別を判定する

## 要素分解の基準

| 種別 | 判定基準 | 成果物への影響 |
| --- | --- | --- |
| 判断ルール | 「〇〇のとき△△する」という条件つき意思決定 | Skill の Gate / 承認フローに反映 |
| 手順 | 順序が重要な操作ステップ | runbook の詳細手順に反映 |
| 注意点 | 違反すると問題が起きる制約・前提 | チェックリストに反映 |
| 指標 | 数値・状態で測れる確認基準 | 完了条件・KPI に反映 |

## 重複検出の判定フロー

```text
類似 Skill あり？
  → YES: 重複度は？
      → 高重複: マージ候補
      → 部分重複: 既存改善候補
  → NO: 再利用価値あり？
      → YES: 新規生成候補
      → NO: スコープ外（ライブラリ対象外）
```

## 操作種別の確定基準

| 操作 | 必須条件 |
| --- | --- |
| 新規生成 | 類似 Skill なし、2チーム以上で再利用見込みあり |
| 既存改善 | 既存 Skill の本文修正で対応可能 |
| マージ | 2件以上の Skill が同一問題を扱い、統合が合理的 |
| 削除 | 陳腐化・重複・90日未参照のいずれかに該当 |
